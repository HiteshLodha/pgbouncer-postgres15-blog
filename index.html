PgBouncer + PostgreSQL 15:
The Silent Breaking Change
Nobody Told Us About

Production Incident Deep-Dive  ·  2025  ·  SRE Engineering

Everything was running smoothly. PgBouncer and PostgreSQL had been humming along in production for years. Then one routine password reset later — database connections started failing across the board. No deployment. No config change. Just a password reset.

This is a story about a deceptive class of production failures: changes that appear completely safe, but leave a hidden tripwire that triggers weeks or months later. Here is the full breakdown of what happened and how you can prevent it.

The migration did not break it. Resetting the password on PostgreSQL 15 — without knowing about a silent default change — did.

The Architecture

A standard pattern: PgBouncer connection pooler in front of a Cloud SQL PostgreSQL instance on GCP. Applications connect through PgBouncer, which pools and multiplexes connections efficiently — preventing PostgreSQL from spawning a new OS process per client.

// Connection flow
App Pods
→
PgBouncer :6432
→
Cloud SQL PG :5432
① App authenticates to PgBouncer (MD5)
② PgBouncer authenticates to PostgreSQL (MD5)
③ PostgreSQL checks pg_shadow — hash format must match auth method

Critical detail: the PgBouncer in use was a standard managed image — no access to pgbouncer.ini or userlist.txt. The image is sealed. This constraint becomes the central challenge when things break.

How PgBouncer Auth Works

PgBouncer has two separate auth interactions. First, it authenticates incoming app connections against its own userlist.txt using the auth_type in pgbouncer.ini. Second, it connects to PostgreSQL as a service account — also using that same auth_type. This backend connection is what broke.

// pgbouncer.ini — key setting
[pgbouncer]
auth_type         = md5    ← PgBouncer speaks MD5 to PostgreSQL
auth_file         = /etc/pgbouncer/userlist.txt
pool_mode         = transaction
max_client_conn   = 200

With auth_type = md5, PgBouncer sends an MD5 hash to PostgreSQL. PostgreSQL checks the stored hash format in pg_shadow. If the stored format is MD5 — connection succeeds. If stored format is SCRAM — mismatch, connection rejected.

MD5 vs SCRAM-SHA-256

MD5 computes MD5(password + username) — simple, stateless, universally supported. No salt. No replay protection. Cryptographically weak by modern standards.

SCRAM-SHA-256 is a 4-step challenge-response protocol (RFC 5802). The password never travels over the wire. Uses PBKDF2 with a random salt and iteration count. Provides mutual authentication. Far more secure — but requires both client and server to support it.

How the Default Changed Across Versions

The password_encryption parameter controls how new passwords are stored. PostgreSQL changed its default at PG 14. Anyone migrating from PG 12 directly to PG 15 crossed this boundary in a single jump — making the change especially invisible.

PG 10
SCRAM introduced (optional)
PG 12
Default: MD5
PG 14
Default changed to SCRAM
PG 15
Default: SCRAM (standard)
PG 16+
Default: SCRAM (enforced)
// Check what your instance uses
SHOW password_encryption;
-- PG 12:  md5
-- PG 15:  scram-sha-256   ← silent change after migration

The Incident Timeline

// Production Timeline
2021
Initial Setup — PostgreSQL 12 Password set on PG 12. Default: MD5. Hash stored as MD5 in pg_shadow. PgBouncer auth_type = MD5. Perfect alignment. System runs smoothly for years.
Jan 2025
Migration — PG 12 → PG 15 Instance upgraded to PG 15. Existing password hashes carried over as-is — still MD5. PgBouncer continues working fine. Migration declared successful. Nobody notices the default has changed.
Post-migration
Hidden Time Bomb Ticking System runs smoothly on PG 15 for weeks. Old MD5 hash still works. Everything looks green. But SHOW password_encryption now returns scram-sha-256. The trap is set.
2025 ⚠️
Routine Password Reset — Connections Fail Service account password reset on PG 15. PG 15 applies its default: SCRAM-SHA-256. New hash stored as SCRAM. PgBouncer sends MD5 auth → PostgreSQL sees SCRAM hash → format mismatch → connection rejected. Incident declared.

Why the Migration Looked Safe

When PostgreSQL upgrades to a new major version, it carries existing pg_shadow hashes over byte-for-byte. It does not rehash them. The old MD5 hash from 2021 survived perfectly on PG 15 — until a password reset forced a new hash to be written using the new default.

// pg_shadow — before vs after password reset on PG 15
-- Right after migration (PG 15 running, old hash intact):
 bouncer_user | md5a1b2c3d4e5f67890...   ← PgBouncer happy ✅

-- After password reset on PG 15 (new default kicks in):
 bouncer_user | SCRAM-SHA-256$4096:...  ← PgBouncer fails ❌
⚠️

The Hidden Time Bomb
After a major PG upgrade across the 13/14 boundary, the system appears healthy — old hashes still work. But the moment anyone resets a password, it gets stored in SCRAM. If your connection pooler only speaks MD5, that is when everything breaks.

Why You Cannot Always Fix PgBouncer

The obvious answer is: update PgBouncer to use SCRAM. Modern PgBouncer (1.16+) fully supports it. But when you are on a standard sealed image — a sidecar from a platform team or managed service — there is no access to its config files. No SSH. No file edits. The image is read-only by design.

💡

Accepted SRE Pattern
When you cannot change the middleware, adapt the database to the middleware's constraints. This is not a hack — it is the pragmatic decision when you only have one lever. Use it.

The Fix: Step-by-Step

Two steps. Both mandatory. Skipping either one means the problem returns on the next password reset.

Step 1 — Force MD5 on the PostgreSQL Instance

// Via gcloud CLI
gcloud sql instances patch YOUR_INSTANCE \
  --database-flags password_encryption=md5

Or via Cloud Console: Cloud SQL → Edit → Flags → Add → password_encryption = md5 → Save.

Step 2 — Reset the Password After the Flag is Applied

// Rewrite the hash as MD5
ALTER USER bouncer_user WITH PASSWORD 'your_password';
-- Must run AFTER the flag is active — flag alone does not rehash existing passwords
🔴

Step 2 is not optional. The flag only affects future passwords. The existing SCRAM hash stays in pg_shadow until you explicitly reset the password. Always do both steps in order.

Step 3 — Verify

// Confirm hash format
SELECT usename, passwd FROM pg_shadow WHERE usename = 'bouncer_user';

-- ✅ Good:  bouncer_user | md5a1b2c3d4e5f67890...
-- ❌ Bad:   bouncer_user | SCRAM-SHA-256$4096:abc...  ← Step 2 not done

Pre-Emptive Checklist

  • Run SHOW password_encryption; on every PG 14+ instance — if it returns scram-sha-256, assess whether all clients support SCRAM
  • Audit hash formats: SELECT usename, left(passwd, 25) FROM pg_shadow; — look for SCRAM-SHA-256$ on service accounts used by MD5-only clients
  • After any major PG version upgrade, set password_encryption explicitly before allowing any password resets
  • Document your PgBouncer image type (sealed vs custom) — this defines your options during incidents
  • Add a pg_shadow hash format check to every post-migration validation checklist
  • If upgrading PgBouncer to 1.16+, migrate in correct order: update PgBouncer first, then switch the DB default, then reset passwords

The Broader SRE Lesson

This incident belongs to a tricky failure category: delayed-impact changes. The root cause was introduced during the migration. But it caused no immediate failure — no alerts, no broken tests. The system looked healthy for weeks.

The failure only materialized when a completely unrelated task triggered the hidden mismatch. From the perspective of whoever reset the password, they changed nothing architectural. From the system's perspective, they had just pulled the pin on a grenade that had been sitting there since January.

✅

Key SRE Instinct
When something breaks after a routine operation — not after a major change — ask: what changed silently during the last major change that only now manifested? The root cause is almost always hiding in that gap. Look backwards, not just at the most recent event.

A migration that appears successful is not the same as a migration that has been fully validated. Part of completing a major version upgrade is explicitly asking: "What assumptions has this system been running on, and are any of them now silently false?"

The Core Lesson

The migration looked safe because old hashes were preserved. But every password reset after the upgrade silently used the new default. Check your defaults. Document your constraints. Never assume a smooth migration means nothing changed under the hood.

Frequently Asked Questions

Does setting password_encryption = md5 affect existing passwords?
No. The flag only affects passwords set or reset after it is applied. Existing hashes in pg_shadow are untouched — which is exactly why Step 2 (resetting the password after the flag change) is mandatory.
Is using MD5 a security risk?
MD5 is weaker than SCRAM, but on private internal networks with VPC and firewall rules, the practical risk is low. Treat MD5 as a temporary workaround while planning a proper SCRAM migration — not a permanent state.
Is this Cloud SQL-specific or does it affect all PostgreSQL?
Any PostgreSQL 14+ instance — Cloud SQL, RDS, Azure Database, or self-hosted. The password_encryption default change is a core PostgreSQL behavior. Cloud SQL surfaces it as a database flag; self-hosted requires editing postgresql.conf and running pg_reload_conf().
How do I migrate to SCRAM properly?
Correct order: (1) Upgrade PgBouncer to 1.16+ with SCRAM support. (2) Set auth_type = scram-sha-256 in pgbouncer.ini. (3) Set password_encryption = scram-sha-256 on PostgreSQL. (4) Reset all service account passwords. (5) Verify via pg_shadow. Always update the client before changing the server default — never the reverse.
Closing

Incidents from silent default changes are among the hardest to diagnose because the root cause happened long before the failure, and the trigger looks unrelated. The standard SRE instinct — look at what changed most recently — leads you in the wrong direction.

Check your pg_shadow. Run SHOW password_encryption on your instances. Add a hash format audit to your post-migration checklist. The migration will look smooth — the first password reset after it might not, unless you are already prepared.
